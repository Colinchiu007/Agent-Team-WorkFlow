# 🔄 跨角色互评协议（增强版）

> **增强说明**：2026-05-13 新增「项目监督机制」，解决 CTO 工作失察问题。
> 根因：CTO 独自开发时无人主动监督，只有用户投诉才被发现。
> 改进：CEO 主动监控里程碑，而非被动等待问题上报。

---

## 一、项目监督机制（新增）

### 1.1 监督责任归属

**CTO 职责**：主动报告，而非被动等待询问。
**CEO 职责**：主动监控里程碑，而非被动等用户投诉。

> 这不是 CTO 的个人问题，是系统的设计缺陷——没有人在主动盯着 CTO 的产出。

### 1.2 里程碑自动上报规则

每完成一个阶段，CTO **必须**写入：
```
team/tasks/PROJECT-001-HOT-SKILL.md
```

| 里程碑 | 触发条件 | 超时未完成 → 自动升级 |
|--------|----------|----------------------|
| **M1: 核心玩法原型** | 盲注系统+×Mult机制可演示 | 超 3 天 → 通知用户 |
| **M2: 商店系统原型** | 购买/刷新小丑牌流程跑通 | 超 3 天 → 通知用户 |
| **M3: 完整游戏可玩版** | 8 Ante 全流程可跑通 | 超 5 天 → 通知用户 |
| **M4: Beta 测试版** | 找 Bug，平衡性调整 | 超 7 天 → 通知用户 |

CTO 每次写入 milestone 后，**自动通知用户**：
> "🎯 PROJECT-001 里程碑报告已产出：M1 核心玩法原型 → 位置：`team/tasks/PROJECT-001-HOT-SKILL.md`"

### 1.3 CEO 主动监控频率

| 检查点 | 频率 | 内容 |
|--------|------|------|
| **每日站会（09:00）** | 每天工作日 | CTO 有无新进展？有无阻塞？ |
| **里程碑倒计时** | 每 24h | 有无 milestone 超时风险？ |
| **周报（周一 09:30）** | 每周 | CTO/PM/COO 全员报告汇总 |
| **阶段评审** | 里程碑完成时 | 用户确认是否进入下一阶段 |

**⚠️ 关键原则**：CTO 的工作有问题 → **CEO 直接通知用户**，不藏着捂着。


### 1.4 升级触发条件（自动通知用户）

| 触发条件 | 通知内容 |
|----------|----------|
| 里程碑超时 3 天以上 | "⚠️ CTO 在 [里程碑名] 停滞 [X] 天，请关注" |
| CTO 报告中有重大技术风险 | "🚨 [风险描述]，建议与 CTO 讨论方案" |
| 游戏核心功能与 PRD 不符 | "❌ PRD 偏差：预期 [X] vs 实际 [Y]，需 CTO 修正" |
| 用户主动提出功能问题 | "📋 用户反馈：[问题]，CTO 需在 24h 内回应" |


> 这不是"打小报告"，是透明沟通——让用户知道 CTO 在做什么、做没做。

## 一、互评触发时机

| 时机 | 触发条件 | 评价对象 | 评价人 |
|------|----------|----------|--------|
| 📅 **周度互评** | 每周周报时 | 所有其他 Agent | CTO + PM + COO |
| ✅ **里程碑互评** | 重要阶段完成时 | 承担该阶段的相关方 | 直接相关的 1-2 个 Agent |
| ⚠️ **专项互评** | 收到对方投诉/冲突时 | 发起方认为有问题的 Agent | 发起方 |
| 📊 **季度复盘** | 每月末 | 所有 Agent | 全体 |

## 二、各角色评价视角矩阵

```
                        ↓ 被评价方
评价方 →    CTO 工作          PM 工作          COO 工作
------------------------------------------------------------
CTO 评价    —（自评）     技术工期合理性     技术可行性/安全性
PM 评价    估算偏差/风险   —（自评）         计划合理性/灵活性
COO 评价   用户体验/协作   运营视角契合度    —（自评）
```

---

## 三、互评流程

```
Step 1: CEO 触发互评
   ↓
Step 2: 各 Agent 读取自己的 Persona 文件（team/personas/[角色]-PERSONA.md）
   ↓
Step 3: Agent 从自己的专业视角审视被评价方的工作
   ↓
Step 4: 各 Agent 写出「评价报告」写入 team/reports/eval-[评价人]-[被评价人]-YYYY-MM-DD.md
   ↓
Step 5: CEO 汇总所有评价 → 写入 team/evaluation/WEEKLY-YYYY-WXX.md
   ↓
Step 6: 识别分歧（如 CTO 说 A，COO 说 B）→ CEO 裁定 → 结果给你
```

## 四、互评报告模板

```markdown
# 互评报告 - [评价人] 评价 [被评价人]

## 基本信息
- 评价人：
- 被评价人：
- 评价时间：
- 评价周期：

## 被评价方本周工作摘要
（被评价方自己提交的报告中，评价人认为值得评价的部分）

## 我的评价（[评价人视角]）
### ✅ 做得好
（从我的专业视角，被评价方做得好的地方）

### ⚠️ 需改进
（从我的专业视角，被评价方需要改进的地方）

### 🚨 重大关切
（如果不改进会影响整体目标的关键问题）

## 具体建议
（可操作的改进建议，不是泛泛而谈）

## 综合评分（从我的视角）
- 工作质量：⭐⭐⭐⭐⭐ / 5
- 协作配合：⭐⭐⭐⭐⭐ / 5
- 时效履行：⭐⭐⭐⭐⭐ / 5
- **我的视角评级：A / B / C / D**

## 我和被评价方的分歧（如有）
（记录评价分歧，但不对抗，由 CEO 最终裁定）
```

## 五、评价冲突处理

### 场景 1：两个 Agent 给出截然相反的评价

> CTO 给 COO 打 D（技术风险），COO 给 CTO 打 C（用户体验）

**处理方式**：
1. CEO 读取双方完整报告
2. CEO 判断分歧点是否影响核心目标
3. 如果影响 → 上报你裁断
4. 如果不影响 → CEO 记录分歧，两方各打各分，取平均或取低

### 场景 2：某人连续多周期评分低

**处理方式**：
1. CEO 在下一周期前主动与低分 Agent 沟通：「根据上周期评价，你在哪方面有困难？」
2. 如困难合理 → 调整任务分配
3. 如困难不合理 → 上报你，建议调整角色职责

## 六、Persona 读取指令

每个 Agent 在进行互评前，必须先读取：

```
读取 team/personas/[角色]-PERSONA.md
→ 了解自己的价值观、评价标准、沟通风格
→ 从这个独特视角出发进行评价
→ 不要脱离自己的 Persona 去评价
```

---

## 七、评价结果应用

| 评级 | 对应措施 |
|------|----------|
| 连续 2 周期 A | 该 Agent 获得更多自主决策权（CEO 减少干预） |
| 出现 B | 正常，发一条改进建议即可 |
| 出现 C | CEO 与该 Agent 单独沟通，产出一份「改进计划」 |
| 连续 2 周期 C 或出现 D | CEO 上报你，建议调整职责或重新定义角色 |
